# Plan: Make Prism Proxy Work Seamlessly with Claude Code

## Context

Prism is an Anthropic ↔ OpenAI protocol bridge. Claude Code uses the `@anthropic-ai/sdk` to communicate with the Anthropic API via strict SSE streaming. The user is experiencing context loss, garbled output, tool call failures, and streaming breaks when using Prism with Claude Code.

After analyzing both codebases, I identified **12 critical issues** across 4 categories that cause these failures.

---

## Issue 1: Thinking Blocks Missing Signatures

**File**: `prism/translate/stream.py` (line ~130)
**File**: `prism/providers/openai_compat.py` (line ~277)

Claude Code requires every `thinking` content block to have a `signature` field. When streaming, `content_block_start` for thinking blocks must include `"signature": ""` and a `signature_delta` event must be emitted at `content_block_stop`. Prism currently emits thinking blocks with empty signature but never emits `signature_delta`.

**Fix**:
- In `_AnthropicStreamBuilder.thinking()`: add `"signature": ""` to `content_block_start` content_block (already done)
- In `_AnthropicStreamBuilder.close_open()`: when closing a thinking block, emit a `signature_delta` event before `content_block_stop`
- In `openai_compat.py translate_response()`: add `"signature": ""` to thinking blocks (already done)

---

## Issue 2: tool_use content_block_start Missing `input` Field

**File**: `prism/translate/stream.py` (line ~156, ~188)

Claude Code's stream parser expects `content_block_start` for `tool_use` blocks to include `"input": {}`. Prism currently emits it correctly in some paths but the `_open()` method at line 104 doesn't always include it.

**Fix**:
- Ensure all `tool_use` `content_block_start` events include `"input": {}` in the content_block (already correct in current code — verify)

---

## Issue 3: message_start Usage Missing Cache Token Fields

**File**: `prism/translate/stream.py` (line ~231)

The `message_start` event's `usage` object must include `cache_creation_input_tokens` and `cache_read_input_tokens` fields. Claude Code's SDK parses these and will crash or produce wrong accounting if missing.

**Current code** (line 241-246): Already includes these fields set to 0. ✅

---

## Issue 4: message_delta Missing `stop_sequence` Field

**File**: `prism/translate/stream.py` (line ~335)

The `message_delta` event must include `stop_sequence` (can be null). Claude Code reads this field.

**Current code** (line 337): Already includes `"stop_sequence": None`. ✅

---

## Issue 5: Thinking Blocks Dropped During Request Translation

**File**: `prism/providers/openai_compat.py` (line ~141-144)

When Claude Code sends conversation history with assistant messages containing `thinking` blocks, Prism drops them silently. This causes context loss because the model sees a gap in the conversation.

**Fix**:
- In `_translate_messages()`: when processing assistant messages with `thinking` blocks, preserve them as text or a recognizable format so the downstream provider can handle them
- For OpenAI-compat providers that support reasoning: map `thinking` → `reasoning_content` in the assistant message
- For providers that don't: concatenate thinking text into the assistant message content with a marker

---

## Issue 6: `thinking` Request Parameter Not Translated

**File**: `prism/providers/openai_compat.py` (line ~49-90)

Claude Code sends `"thinking": {"type": "adaptive"}` or `"thinking": {"type": "enabled", "budget_tokens": N}` in the request body. Prism ignores this entirely, so providers that support reasoning (DeepSeek, etc.) never get activated.

**Fix**:
- In `translate_request()`: detect `thinking` parameter and map to provider-specific fields
- For OpenAI-compat: map to `reasoning_effort: "high"` or similar if the provider supports it
- Log a warning if thinking is requested but provider doesn't support it

---

## Issue 7: `cache_control` Markers Dropped from Content Blocks

**File**: `prism/providers/openai_compat.py` (line ~130-220)

Claude Code sends `cache_control: { type: "ephemeral" }` on content blocks for prompt caching. Prism drops these during translation, which means:
1. The provider never gets cache hints
2. When round-tripping conversation history, cache markers are lost

**Fix**:
- In `_translate_messages()`: strip `cache_control` from blocks (already done — providers don't support it)
- This is acceptable for OpenAI-compat providers since they don't have prompt caching

---

## Issue 8: `tool_choice` `disable_parallel_tool_use` Not Fully Translated

**File**: `prism/providers/openai_compat.py` (line ~224-238)

Claude Code sends `"tool_choice": {"type": "auto", "disable_parallel_tool_use": true}`. Prism translates `type` but may not handle `disable_parallel_tool_use` in all code paths.

**Current code** (line 80-81): Already handles this. ✅

---

## Issue 9: Beta Headers Passed Through But Not Stripped for OpenAI-Compat

**File**: `prism/proxy.py` (line ~176-180)

Claude Code sends many beta headers (`thinking-2024-11-20`, `prompt-caching-2025-07-01`, etc.). Prism passes them to OpenAI-compat providers, which may reject unknown headers.

**Fix**:
- In `_handle_request()`: strip all `anthropic-beta` headers before sending to OpenAI-compat providers
- Keep them only for Anthropic-native passthrough providers
- Log which betas were stripped for debugging

---

## Issue 10: Error Response Format Incomplete

**File**: `prism/translate/errors.py`

Claude Code checks for specific error shapes. The current error translation is mostly correct but missing:
- `retry_after` header in rate-limit responses
- Proper `request-id` in error responses

**Fix**:
- Add `retry_after` extraction from provider `Retry-After` headers
- Ensure all error responses include a unique `request-id`

---

## Issue 11: Non-Streaming Response Missing Fields

**File**: `prism/proxy.py` (line ~396-405)

The non-streaming response is missing:
- `stop_sequence: null` (already present ✅)
- `usage.cache_creation_input_tokens` and `usage.cache_read_input_tokens` (need to add)

**Fix**:
- Add `cache_creation_input_tokens: 0` and `cache_read_input_tokens: 0` to usage in non-streaming responses

---

## Issue 12: Stream Builder Doesn't Handle `redacted_thinking` Blocks

**File**: `prism/translate/stream.py`

Claude Code can send `redacted_thinking` blocks in conversation history. The translator doesn't handle these, which could cause issues when round-tripping.

**Fix**:
- In `_translate_messages()`: handle `redacted_thinking` blocks by preserving them as-is or converting to a recognizable format
